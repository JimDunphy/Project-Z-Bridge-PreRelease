# Project Z-Bridge — Status / Handoff

Goal: run the **classic Zimbra web client** against a **Stalwart Mail Server (JMAP)** by translating `/service/soap` calls into JMAP.

## How to run (dev)

- Put `zimbra.war` (from:zm-web-client.git) in the repo root (gitignored).
  - Simplest current acquisition path if you do not already have one: `./manage.sh webclient-build`
  - This downloads the latest supported `zimbra.war` from `JimDunphy/DockerZimbraRHEL8` GitHub Releases into the repo root.
  - If you want a custom or self-built WAR instead, use `DockerZimbraRHEL8` manually with `./docker.sh --build-war 10.1`.
- Extract web assets: `./manage.sh webclient-extract` → `static/zimbra/` (gitignored).
- Configure upstream JMAP creds in `.env` (copy from `.env.example`; do not commit secrets).
- Start: `./manage.sh up` (preferred; ensures `HOST_UID/HOST_GID` are set so Cargo can write to `shim/target/` inside the container)
- If you suspect old containers are still running old code: `./manage.sh restart` (does `down` + `up --build`)
- Open: `http://127.0.0.1:${BRIDGE_PORT:-7777}/zimbra/`

Ports: docker-compose binds `127.0.0.1:${BRIDGE_PORT:-7777}:7777` so it only listens on localhost by default (`docker-compose.yml:1`).

## How to run (runtime image)

- Build a release image: `./manage.sh image-build` (or load a prebuilt image tar via `docker load`).
- Extract web assets: `./manage.sh webclient-extract` → `static/zimbra/`.
- Start: `./manage.sh up-runtime` (uses `docker-compose.runtime.yml`).
- Logs: `./manage.sh logs-runtime`.

## What works now

- A number of endpoints and mappings are implemented (Auth, folder tree, basic mail listing, message viewing, core message actions, and the attachment/blob proxy plumbing).
- Mail folder creation: `CreateFolderRequest` is implemented for mail folders (`view=message`) via JMAP `Mailbox/set` and emits `Header.context.notify.created.folder` so the folder tree updates without reload (please verify by creating a nested folder like `Archives/Later`).
- Folder properties (mail folders): `FolderActionRequest op="rename"` is implemented via JMAP `Mailbox/set`, and `op="color"` is implemented via shim-side storage (`BRIDGE_DATA_DIR/folders/*`) + `notify.modified.folder` updates (please verify rename + color persist after refresh).
- Mail folder moves/deletes: `FolderActionRequest op="move"` updates mailbox `parentId` via JMAP `Mailbox/set`; `op="trash"` and `op="delete"` move folders under Trash, and `op="delete"` on a folder already under Trash permanently destroys the mailbox via JMAP `Mailbox/set destroy` (needs browser verification).
- Shared mail folders (internal-only): shared mailboxes are discovered from Stalwart mailbox ACLs (`Mailbox.shareWith`) and exposed to ZWC as mountpoints (`folderTree.link`), including list/open/read and core message actions scoped to the shared account. ZWC share grants/revokes are mapped to Stalwart mailbox ACL (`shareWith`) and share-notification email is sent via JMAP Submission. v1: auto-visible (no accept/mount flow), tagging disabled. Needs browser verification; see `docs/developer/SHARING_SHARED_FOLDERS.md`.
- RSS/Atom feed folders: ZWC “Subscribe to feed” creates a normal folder with `folder.url`; the shim validates/fetches/parses the URL at creation, stores `url` in folder metadata, and imports an initial batch of items immediately (unread). `FolderActionRequest op="sync"` forces refresh, and NoOp-driven polling can auto-import new items as unread so folder counts reflect updates. Includes SSRF guardrails, redirect limits, and response size caps; see `docs/developer/RSS_FEEDS.md` and `docs/developer/RSS_FEEDS_POSTMORTEM.md`.
- Verified in UI (2025-12-17): folder tree renders; opening a message shows the full body content.
- Mail read-state persistence: ZWC marks messages read via `GetMsgRequest.m.read=1`. The shim now applies that to upstream JMAP `$seen` so read state persists when switching folders/refreshing (needs verification in your browser).
- Mail HTML compatibility: ZWC defaults `zimbraPrefMessageViewHtmlPreferred` to `FALSE` unless the server returns it in `GetInfoResponse`, which causes HTML mail to render in the plain-text view (no clickable links, no “Display Images” bar). The shim now returns `zimbraPrefMessageViewHtmlPreferred=TRUE` by default, ensures external links get `target="_blank"` (Firefox/Gecko), defangs external images (`src` → `dfsrc`), and serves `/img/*` so the placeholder `/img/zimbra/1x1-trans.png` does not 404. Verified via Playwright; please confirm in your browser (see `BUGS.md`).
- Mail reading-pane first paint: when the user preference path is `MARK_READ_NOW`, the conversation reading-pane header now renders immediately in the read presentation instead of flashing from unread to read a fraction of a second later. Important implementation note: this was fixed at the header-render layer (`ZmMailMsgCapsuleViewHeader.set()` path), not by mutating message/conversation unread state early; earlier broad state mutations broke list behavior. Verified live on `user@example.com` on 2026-04-20.
- Mail reading-pane layout: the bridge now left-anchors common centered newsletter wrappers in the same-origin message iframe using structure-based heuristics (for example, empty spacer-cell tables or narrow centered wrappers with `align="center"` / auto margins / `max-width`). This improves wide-pane reading behavior for messages like `Too many mental calories` and `Oil Jumps`. It is intentionally not a full responsive-email rewriter yet; narrower-pane shrink/reflow remains future enhancement work and is tracked in `BUGS.md`.
- Mail link affordance in HTML bodies: some senders emit anchors that survive sanitization but inline-reset themselves with `color: unset; text-decoration: unset`. The bridge now restores visible hyperlink affordance for those affected **text** links after render, while deliberately skipping image links and button-style anchors. Verified live on the `Subscribe here` link in `Too many mental calories` on 2026-04-20.
- Themes (skins): `GetAvailableSkinsRequest`/`GetAvailableLocalesRequest`/`GetAvailableCsvFormatsRequest` are implemented so the Preferences UI can populate its dropdowns. The default theme can be set via `BRIDGE_DEFAULT_SKIN`, and ZWC theme changes persist via `ModifyPrefsRequest zimbraPrefSkin` (stored per-user in `BRIDGE_DATA_DIR/prefs/*` and reflected in `GetInfoResponse.prefs`).
- Preferences → Import/Export: ZWC’s Import/Export controller is wired up via `/home` REST endpoints — import is `POST /home/<user>/<path>?fmt=csv|ics|tgz&callback=...` (multipart) and export is `GET /home/<user>/<path>?fmt=csv|ics|tgz`. `.tgz` uses a Project Z Bridge v1 archive layout (`contacts/contacts.csv`, `calendar/calendar.ics`, `mail/index.json`, `mail/messages/*.eml`) and imports mail via JMAP `Email/import`. Account export is streamed so downloads start immediately; if mail export fails mid-run, the archive includes `mail/ERROR.txt`.
- Import compatibility detail: `.tgz` import supports Project Z Bridge v1 archives and Zimbra contacts-only `.tgz` exports. Zimbra calendar/account `.tgz` exports (`*.appt` / `*.appt.meta`) are not supported yet and now return an explicit unsupported-format error instructing users to use `.ics` for calendar import.
- Calendar ICS import now preserves yearly recurrence via JMAP `recurrenceRule` (singular) and honors `resolve=replace` / `ImportAppointmentsRequest.resolve=replace` by matching existing events on `uid` and updating them in place. The bridge calendar search/minical paths also expand yearly recurrences locally instead of relying on upstream `CalendarEvent/query after/before`, which does not surface future recurring instances. Verified live on 2026-03-15 with `Birthdays-2026-03-15-172427.ics`: `user B-Day` appears on `2026-04-21` in both `SearchRequest types=appointment` and `GetMiniCalRequest`.
- Calendar import/color repaint: successful `ImportAppointmentsRequest` and calendar-folder color changes now trigger a full active-calendar view rebuild in the client, so imported items and recolored appointments appear immediately instead of requiring a manual refresh. Verified live on 2026-03-15 with `Dr Virk Meet & Greet` from `cal-export/Dunphy Family Events-2026-03-15-213914.ics`; the imported March 12 event rendered in Month view and repainted when the `Calendar` folder color changed.
- Calendar ICS import now also tolerates several real-world Zimbra export edge cases: events missing explicit `DURATION`/`DTEND` (derives from `utcStart`/`utcEnd`, `start`/`end`, or falls back to `P1D` for all-day and `PT0S` for timed start-only entries), duplicate `uid`s inside a single `.ics` (dedupes/merges before `CalendarEvent/set`), and `resolve=replace` now matches existing `uid`s across the account instead of only inside the target calendar so earlier repair imports can be moved cleanly into their final folders. Verified live on 2026-03-15 by importing every file under `cal-export/` into `user@example.com` (`AesirDomains`, `user Calendar`, `user Family Activities & Events`, `Birthdays`, `Calendar`, `Saanich Recycle Bins`, `Saanich Refuse`) and spot-checking rendered April 2026 month-view data for `user B-Day`, `Recycling`, `Garbage cart and organics`, and `Betsy & Jim Dunphy | Schwab Wealth Advisory Meeting`.
- Calendar REST `.ics` import now matches duplicate `uid`s across the full account and defaults classic ZWC imports to replace-by-UID even though the UI exposes no duplicate-policy control. Calendar delete/trash for non-default calendars now hard-deletes immediately instead of leaving hidden ghost calendars under `Deleted Items`. Verified live on 2026-03-16 by deleting `Dunphy Family Events`, recreating it, importing `cal-export/Dunphy Family Events-2026-03-15-213914.ics`, deleting it again, and re-importing successfully; the recreated visible calendar contained `Dev - cartrophen` (`UID 00439FF9-F653-4EF2-926B-A2844E1AA636`) and `Dr Virk Meet & Greet`.
- Calendar REST `.ics` import now reloads the live classic ZWC organizer tree after success and immediately restores the imported calendar into checked state. The stock iframe callback only displayed `Import succeeded` and sent `NoOp`, which left the current browser session stale even though upstream JMAP state and bridge `GetFolderRequest` were already correct. Verified live on 2026-03-16 in the same `user@example.com` session: recreated `Dunphy Family Events` now appears immediately in the left calendar tree without a hard refresh.
- Calendar REST `.ics` import refresh no longer uses timer bursts. The final stable client path is: reload folders once, restore checked state, clear the stale Month grid, then force a strong calendar controller refresh. Calendar import/delete also now syncs `calendar_event_state` immediately so later `NoOpRequest` polls do not replay stale `CalendarEvent/changes` pages and repaint the month view a second time. Final verified behavior on 2026-03-16: after import, Month view can stay blank for roughly `4s`-`5s`, then repaints once with correct single entries and no duplicate overlays. Full debugging trail: `docs/developer/CALENDAR_IMPORT_VIEW_DEBUGGING.md`.
- Calendar display performance: month/day views now use a shared per-session raw-event cache plus a shared window-occurrence cache, with adjacent-window prewarm, so hot calendar navigation is typically in the `0.16s`-`0.36s` range instead of multi-second repeated sweeps. Verified live on 2026-03-16 via bridge logs and browser XHR timing (`BatchRequest` hot path measured at `165ms`, `338ms`, `345ms`, `346ms`, `349ms`, `356ms`; browser-side `BatchRequest` timings measured `350.9ms`, `356.1ms`, `343.6ms`, `370.7ms`).
- Calendar cold-start note: the first calendar open after login/restart is still the expensive path because classic ZWC issues overlapping cold bootstrap requests (`GetMiniCalRequest` plus calendar `SearchRequest`, sometimes both batched and standalone) before caches are warm. Verified live on 2026-03-16 with cold timings of roughly `2.1s`, `4.2s`, and `4.9s` depending on the exact request shape. This is now considered acceptable; any future optimization should target only first-open prewarm, not the hot month-hop path.
- Calendar cache safety note: the new caches are read-side only and are invalidated on calendar mutations. Expected risk is low for import/share correctness and limited mainly to UI freshness if an invalidation edge is missed. Watch specifically for: import completes but items are not visible until refresh; shared-calendar checkbox/state changes not reflecting immediately; recurring edits not appearing until the next navigation.
- Preferences → General → Language/Time Zone: `GetAvailableLocalesRequest` enumerates locales from extracted ZWC message bundles (`static/zimbra/WEB-INF/classes/messages/ZmMsg_*.properties`), and `ModifyPrefsRequest` persists `zimbraPrefLocale` + `zimbraPrefTimeZoneId` into `BRIDGE_DATA_DIR/prefs/*` (needs user verification in browser).
- Preferences → General → Appearance: `ModifyPrefsRequest` persists appearance defaults (`zimbraPrefComposeFormat`, `zimbraPrefHtmlEditorDefaultFontFamily`, `zimbraPrefHtmlEditorDefaultFontSize`, `zimbraPrefHtmlEditorDefaultFontColor`, `zimbraPrefFont`, `zimbraPrefFontSize`, `zimbraPrefDefaultPrintFontSize`) in `BRIDGE_DATA_DIR/prefs/*` and reflects them in `GetInfoResponse.prefs`.
- Preferences → General → Other Settings → Bridge Safety: the bridge now injects two managed safety features into the General prefs page:
  - `Tag likely phishing email`
  - `Tag foreign-country email`
  These persist as bridge-managed mail prefs (`zimbraPrefZBridgePhishTaggingEnabled`, `zimbraPrefZBridgeForeignCountryTaggingEnabled`) plus tag-color prefs (`zimbraPrefZBridgePhishTagColor`, `zimbraPrefZBridgeForeignCountryTagColor`). When enabled, the shim ensures the managed tags exist (`Likely Phish`, `Foreign Country`), reconciles their configured colors, and compiles hidden delivery-time Sieve rules **ahead of ordinary user filters** so ZWC does not expose them as normal editable filters. Current signals are:
  - phishing: `X-Spamd-Result contains J_PHISH_HIGH(`
  - foreign-country: `X-Spamd-Result contains BLACKLIST_COUNTRY(` by default; the opening `(` is recommended because the hidden Sieve rule uses simple raw-header `contains` matching against the usual Rspamd `SYMBOL(score)` form. Override via `BRIDGE_MANAGED_FOREIGN_COUNTRY_SYMBOLS` if your deployment emits a different symbol shape.
  Current UI uses stable native `<select>` dropdowns for colors; a compact native-style tag-color picker was attempted and rolled back after it destabilized the client shell (see `BUGS.md` / `docs/developer/DEV_DEBUGGING.md`).
- Preferences → Mail → Trusted Addresses: `ModifyPrefsRequest zimbraPrefMailTrustedSenderList` persists the allowlist in `BRIDGE_DATA_DIR/prefs/*` and returns it in `GetInfoResponse.prefs`. When the sender matches the allowlist, message rendering skips external-image defanging so external images load automatically (still sanitizes HTML).
- Preferences → Signatures: `CreateSignatureRequest` / `ModifySignatureRequest` / `DeleteSignatureRequest` are implemented via the shim’s `BridgeStore` (default: `BRIDGE_DATA_DIR/signatures/*`) and returned in `GetInfoResponse.signatures`. ZWC will not insert a signature into compose unless you set the “Using Signatures” defaults (New Messages / Replies & Forwards), which triggers `ModifyIdentityRequest` to persist `zimbraPrefDefaultSignatureId` / `zimbraPrefForwardReplySignatureId` (default store: `BRIDGE_DATA_DIR/identities/*`).
- Preferences → Contacts: `ModifyPrefsRequest` persists contact UI prefs (`zimbraPrefAutoAddAddressEnabled`, `zimbraPrefAutoCompleteQuickCompletionOnComma`, `zimbraPrefShareContactsInAutoComplete`, `zimbraPrefSharedAddrBookAutoCompleteEnabled`) in `BRIDGE_DATA_DIR/prefs/*` and reflects them in `GetInfoResponse.prefs`. The shim ensures an `Emailed Contacts` address book exists upstream (created on first login if missing). Auto-add to `Emailed Contacts` is enabled by default (ZWC default) and can be disabled in Preferences. When enabled, `SendMsgRequest` will best-effort add new recipients to `Emailed Contacts` via JMAP Contacts.
- Compose recipient autocomplete: `AutoCompleteRequest` returns `AutoCompleteResponse.match[]` by using a per-session cached contact list and local **substring** matching across display name + first/last + email. This avoids Stalwart’s current token-based `ContactCard/query filter.text` behavior (which will not match partial strings like `vin` for “Vincent”). The first autocomplete request after login may be slower while the cache is populated; subsequent lookups should be instant. Verified by user (2026-01-21).
- Tags: `GetTagRequest`/`CreateTagRequest`/`TagActionRequest` (create/rename/color/delete) + message `tag`/`!tag` actions are implemented via shim-side storage + JMAP keywords; `tag:"Name"` searches map to JMAP `hasKeyword`. Saved Searches is still display-only. Colored tag icons depend on `GET /zimbra/public/loadImgData.jsp` (fixed a JS parse error); please verify tag colors render in your browser. Tag-column sorting is supported via a custom `sortBy` (`tagNameAsc`/`tagNameDesc`) mapped to JMAP keyword presence in tag-name order (needs verification in your browser).
- Verified locally (Playwright + API smoke tests; needs user confirmation in a real browser): styles load (CSS token expansion), login completes, Calendar view loads with calendars checked by default, and appointments render from Stalwart.
- Mail compose plumbing: `SaveDraftRequest` (draft) + `SendMsgRequest` (send) are implemented via JMAP (`Email/set` then `EmailSubmission/set`) and include outbound attachments via `POST /service/upload` + reply threading (`inReplyTo`/`references` from `origid`). Draft auto-save now **replaces** the current draft when ZWC provides `m.id`/`m.did` (create new + destroy previous) so Drafts doesn’t accumulate duplicates (Stalwart doesn’t support in-place draft body updates). A successful send best-effort destroys the referenced draft so Drafts doesn’t accumulate. Verified via a JMAP query + Playwright-triggered saves; please confirm in your browser.
- New-window compose launcher: `GET /zimbra/public/launchNewWindow.jsp` is served as HTML (ZWC’s shipped JSP cannot run in this shim) (verified by user: compose window opens).
- Calendar mutations: `CreateAppointmentRequest` / `ModifyAppointmentRequest` / `CancelAppointmentRequest` are implemented and emit SOAP `Header.context.notify` blocks so ZWC refreshes its appointment cache (verified by user: create/edit/delete updates show without refresh, 2025-12-19).
- Calendar editing: ZWC loads appointment details via `GetMsgRequest` (invite message); the shim falls back to `CalendarEvent/get` and synthesizes an invite-like `GetMsgResponse` when `Email/get` returns no message (needs verification in your browser).
- Calendar invites (RSVP): `GetMsgResponse` includes an `inv` node for real `text/calendar` email parts so ZWC renders Accept/Tentative/Decline UI (verified against Office365-style invites and a live Cloudflare Webinars invite on 2026-04-18; see `docs/developer/CALENDAR_INVITES.md`). `SendInviteReplyRequest` updates `CalendarEvent.participants[*].participationStatus`, optionally triggers Stalwart scheduling replies (`sendSchedulingMessages`), emits calendar refresh notifications so the accepted appointment appears in Calendar immediately, and moves the invite message to Trash.
- Calendar editor bootstrap: `GetInfoResponse` now includes minimal `identities` + `signatures` stubs, which ZWC expects (without them the editor can crash) (needs verification in your browser).
- Contacts: REST canonical contacts load (`GET /home/<user>/Contacts?fmt=cf&t=2&all=all`) + SOAP CRUD (`GetContactsRequest`/`CreateContactRequest`/`ModifyContactRequest`) are implemented via JMAP (`ContactCard/*`) and map core ZWC fields (name parts, nickname, company/department/title, emails/phones, addresses, URLs, birthday/anniversary, notes). `SearchRequest` (contacts) honors `inid:` folder targeting so the Contacts Trash view does not mirror Contacts. To match ZWC expectations for the “All/123/A..Z” alphabet bar + partial search terms, the shim fetches the full contact list for the selected address book and filters locally (Stalwart `filter.text` does not match partial prefixes). Contact ids exposed to ZWC are namespaced as `ct-<jmapId>` to avoid collisions with ZWC system folder ids (e.g. Contacts folder id `"7"`). Verified by user (2026-01-21).
- Mail polling: `GetInfoResponse` enables `zimbraFeatureInstantNotify` + sets `zimbraPrefMailPollingInterval` (from `BRIDGE_MAIL_POLL_INTERVAL`, default `500`) and the shim uses numeric session ids so ZWC uses `NoOpRequest wait=1` long-poll; `NoOpRequest` blocks (default 30s; `BRIDGE_NOOP_WAIT_SECS`) and returns `notify.modified.folder[]` counts (from JMAP `Mailbox/get`) plus best-effort `notify.created.m[]` for Inbox when new mail arrives (triggered by an Inbox “head id” change or an Inbox total-count increase fallback). Message summaries include `cid` from JMAP `threadId` so conversation view can update. As of 2026-03-15, `NoOpRequest` also runs a bounded self-heal over recent Inbox mail using the active incoming filter runtime before emitting Inbox create notifications, so list/bulk mail (for example Activity Stream candidates) can be corrected even when delivery-time behavior did not already move it upstream; still needs broader browser verification—see `BUGS.md`. The self-heal path is currently under renewed review after Betsy's April 2026 Junk/Not Spam report; a separate `MsgActionRequest op="!spam"` destination bug was fixed first, and self-heal removal remains a possible future cleanup if delivery-time filtering proves reliable without it.
- Dumpster: env-gated support is available for “Recover deleted items” listing (`SearchRequest inDumpster=1`) and restore (`op="recover"`) using Stalwart management endpoints (`/api/store/undelete`, `/api/store/blobs`). `GetInfoResponse.zimbraDumpsterEnabled` reflects `BRIDGE_DUMPSTER_ENABLED`. Current limitation: destination folder selection is not fully honored yet (Stalwart undelete restores to Inbox).
- Mini calendar UX: ZWC defaults to a 5s delay before creating the left-pane mini calendar (`ZmCalendarApp.MINICAL_DELAY=5000`). The shim overrides this at page-load so the widget should appear immediately; tune via `BRIDGE_MINICAL_DELAY_MS` (default `0`). Verified by user (2025-12-30).
- Calendar refresh: `NoOpRequest` tracks upstream `CalendarEvent` state and emits `notify.created.appt[]` cache-bust notifications when calendar data changes outside ZWC (e.g., `smmailbox clone-calendars`), so Day/Week views update without requiring logout/login (verified by user, 2026-01-16).
- Calendar folder properties: appointment organizers now include `n` totals sourced from cached per-calendar `CalendarEvent/query calculateTotal=true` counts, so `Edit Properties` shows the real `Appointments: N` value instead of `0`. Verified live on 2026-03-15 (`Birthdays` showed `Appointments: 19`).
- Connection resilience: ZWC hides UI errors for polling failures, so the shim injects a small “connection lost” UI (banner → modal with “Retry Now”) for network glitches, and returns `service.AUTH_REQUIRED` + expires `ZM_AUTH_TOKEN` when the bridge restarts (stale cookie), so ZWC shows the login UI instead of staying in a broken/half-connected state.
- Mail-content hardening:
  - message HTML is sanitized server-side before it is handed back to the client
  - external images are defanged by default and, when explicitly allowed, are loaded through the bridge-controlled signed image proxy
  - sanitized message HTML can be rendered through the dedicated `/service/email-renderer/view/:token` path with a stricter enforced CSP (`default-src 'none'; script-src 'none'; connect-src 'none'; object-src 'none'; frame-src 'none'; font-src 'none'; media-src 'none';`)
  - the legacy `/home/*` and `/service/home/*` part-fetch path now forces active-content document types (`text/html`, XHTML, SVG, XML) to download as attachments and adds `nosniff` plus a defensive sandbox CSP to those responses
  - when deployed behind HTTPS, `ZM_AUTH_TOKEN` cookies can and should be marked `Secure` via `BRIDGE_COOKIE_SECURE=1` and/or `BRIDGE_PUBLIC_BASE_URL=https://...`
- Mail list sorting: `SearchRequest.sortBy` is now mapped to JMAP `Email/query.sort` for the common ZWC sorts (date/subject/from/to/size plus read/flag via `hasKeyword`). Unsupported sorts currently fall back to date; please verify header-click sorting in your browser.
- Mail action UI refresh: `MsgActionRequest`/`ItemActionRequest` ops (`read`/`!read`/`flag`/`update` plus move-like ops `delete`/`trash`/`move`/`spam`) emit `Header.context.notify.modified.m[]` (flags and/or destination `l`) so ZWC can repaint without a hard refresh; move-like ops also preserve `$seen` when changing mailboxes (needs verification in your browser).
- Conversation mode plumbing: `GetInfoResponse` advertises conversations enabled, `SearchRequest` supports `types=conversation` (JMAP `collapseThreads`), and ZWC thread expansion/action endpoints (`SearchConvRequest`, `GetConvRequest`, `ConvActionRequest`) are implemented via JMAP threads. The bridge follows upstream `threadId` semantics and does not add extra subject-only grouping; this means some repeated alert/notification messages that old ZWC appeared to merge by subject will remain separate rows when upstream threading says they are unrelated. See `docs/developer/MAIL_THREADING_SEMANTICS.md`.
- Conversation view preference: `ModifyPrefsRequest zimbraPrefGroupMailBy` now persists (`message` vs `conversation`) and `GetInfoResponse.prefs.zimbraPrefGroupMailBy` reflects the stored value.
- Filters: Preferences → Filters is enabled via `zimbraFeatureFiltersEnabled`. Incoming filter rules are stored upstream by embedding the rule model as base64 JSON inside a Sieve script (`# Zbridge-FilterRules: ...` comment lines). If no zbridge model exists yet, the shim will try a best-effort import from the active upstream script (and any included base script) when it uses the marker format (`# rule:[name]` / `# Rule: name`). In docker-compose dev, `BRIDGE_SIEVE_ACTIVATE` defaults to `0` (store model-only under `<incoming>-model`, do not change delivery-time filtering). Set `BRIDGE_SIEVE_ACTIVATE=1` to compile active rules → upload → validate → activate via JMAP `SieveScript/*` on save. Outgoing rules are stored upstream in a non-active Sieve script (execution not implemented). `ApplyFilterRulesRequest` supports the common “id chunk” path (`m ids="..."`) for a small v1 subset (move/fileinto, mark read, flag/star, tag, discard→Trash, stop) plus tests `headerTest`/`headerExistsTest`/`addressTest`/`addressBookTest`/`contactRankingTest`/`meTest`/`listTest`/`bulkTest`/`attachmentTest`/`bodyTest`; see `docs/developer/FILTERS_PLAN.md`. Delivery-time numeric spam-header compile now covers `X-Spam-Score`, `X-Spam-Status` (`score=...`), and `X-Spamd-Result` (first numeric token), which prevents mixed rule chains such as `Activity Stream` followed by `Move Spam to Junk` from silently dropping the later spam rule when Sieve activation is enabled. For real-world reconciliation, the bridge also reuses that same runtime evaluator during `NoOpRequest` for a bounded set of recent Inbox messages, so active supported rules can self-heal misfiled mail without waiting for a manual folder run. Contact-based compile guardrails are enforced by `BRIDGE_FILTER_ADDRESS_SET_MAX` and `BRIDGE_FILTER_SIEVE_SCRIPT_MAX_BYTES` (set either to `0` to disable); contact groups are intentionally not auto-expanded into filter source sets by default.
- Spam allow/block lists: Preferences → Mail → Spam Mail Options (`GetWhiteBlackListRequest` / `ModifyWhiteBlackListRequest`) persist allow/block sender entries in `# Zbridge-MailPrefs:` comments and inject them as transparent rules at the top of the **compiled incoming** Sieve script (block → Junk, allow → Inbox; matches `From` header or envelope-from). Delivery-time behavior requires `BRIDGE_SIEVE_ACTIVATE=1`. This currently expands each entry inline in Sieve; future preferred design is documented in `docs/developer/PREFERENCES_SPAM_EXTLISTS_PLAN.md` and would move delivery-time membership checks to Sieve `extlists`.

## Known issues (reported)

- Mail HTML layout is still heuristic rather than fully responsive. The bridge now left-anchors common centered newsletter wrappers, but adaptive outer-wrapper shrink/reflow for narrow panes remains future work; see `BUGS.md` for the documented follow-up.
- Conversation view is implemented but needs broader verification (pagination, mark-read behavior, and some action edge cases may still differ from a real Zimbra server).
- Timezone dataset now prefers the extracted ZWC `AjxTimezoneData.js` (full list); if assets are missing, the shim falls back to a minimal built-in set (`UTC`, `America/Los_Angeles`, `America/Vancouver`). Browser verification is still pending.
- Calendar: open/edit/delete handlers are implemented (`GetAppointmentRequest` / `ModifyAppointmentRequest` / `CancelAppointmentRequest`). ZWC-facing appointment ids are namespaced as `appt-<CalendarEvent.id>` to avoid collisions with Zimbra system folder ids (Inbox=2, Trash=3, etc); the mapping is reversible and requires no persistent ID vault. There are still UI issues to iron out (layout/naming) (see `BUGS.md`).
- Calendar reminders: in-app popups are driven by `alarmData` derived from Stalwart `CalendarEvent.alerts` (verified by user: reminder popup shows). To avoid “past-due reminder spam” on login, the shim omits `alarmData` for appointments that have already ended, and it tracks per-session dismiss/snooze state so refreshes respect `DismissCalendarItemAlarmRequest`/`SnoozeCalendarItemAlarmRequest` (needs user verification in a real browser). Email reminders are written to Stalwart as `action=email` alerts and delivered server-side; delivery depends on server settings (not yet verified): `calendar.alarms.allow-external-recipients` (default `false`, external recipients overridden to the account) and `calendar.alarms.minimum-interval` (default `1h`, near-term alarms may be skipped). SMS reminders are also stored as `action=email` alerts (attendee is the carrier gateway address) with an `x-zbridge-channel=sms` marker so the UI can round-trip “SMS vs Email”; currently the shim stores only one delivery channel (if both EMAIL and DEVICE_EMAIL are set, it drops DEVICE_EMAIL). SMS device-email setup loads carrier/region data via `/res/ZmSMS.js`, and the verification endpoints (`SendVerificationCodeRequest` / `VerifyCodeRequest` / `InvalidateReminderDeviceRequest`) are implemented (needs verification in browser).
- Out‑of‑Office (vacation): `GetInfoResponse.prefs` exposes ZWC’s OOO prefs and `ModifyPrefsRequest` persists them in the embedded `# Zbridge-MailPrefs:` model + injects a conservative Sieve `vacation :days 7` block into the active incoming script (delivery-time, server-backed; avoids lists/bulk, auto-submitted mail, DSNs, and obvious spam markers). “Internal senders” uses sender-domain == user-domain; `externalSendersType=ALLNOTINAB` is treated as `ALL` for v1 (needs user verification in browser). Bridge extension: per-recipient alias-targeted replies are compiled ahead of global fallback, with browser UI rows injected into Preferences; persistence uses repeated `zimbraPrefOutOfOfficeAliasReply` values and canonical read-back via `zimbraPrefOutOfOfficeAliasRepliesJson`.
- CSP: best-effort Content Security Policy is enabled in report-only mode by default (`BRIDGE_CSP_MODE`/`BRIDGE_CSP`) to constrain external connections while keeping ZWC functional; set `BRIDGE_CSP_MODE=log` to collect violation reports at `BRIDGE_CSP_REPORT_URI` (recommended `/zimbra/csp-report` behind reverse proxies), and flip to enforce mode after confirming no breakage.

See `BUGS.md` for the current list.

## Key implementation points

- SOAP XML support: ZWC often sends `application/soap+xml` requests (including `BatchRequest`). We parse SOAP XML into a Zimbra-style JSON envelope so the existing JSON handlers can be reused (`shim/src/soap_xml.rs:1`).
- Startup stubs required for the client to not crash:
  - `GetMailboxMetadataRequest` returns `meta: [{ section, _attrs: {} }]`.
- `DiscoverRightsRequest` returns `targets: []`.
- `/home` + `/service/home` REST endpoints: implemented for message part/source download (`id=<msgId>&part=<mimePart>` / `view=text`) plus Import/Export formats (`fmt=csv` for contacts, `fmt=ics` for calendar, `fmt=tgz` full mailbox archive; `POST` supports `fmt=csv|ics|tgz` imports). Requests with an unhandled `fmt=...` return a visible iframe callback error (so users don’t silently download empty files).
- `POST /service/upload` is implemented (ZWC compose attachments upload path) and proxies to JMAP `Blob/upload`.
- Folder IDs: Zimbra expects stable numeric-ish folder IDs. We now derive them deterministically from the JMAP mailbox id (restart-safe), while preserving Zimbra’s system folder IDs (Inbox=2, Trash=3, etc.).
- Calendar IDs: Appointment IDs presented to ZWC are `appt-<CalendarEvent.id>`; the shim strips the prefix when talking to upstream JMAP, avoiding collisions with Zimbra folder ids while keeping the mapping reversible without a persistent ID vault.
- Stalwart quirk: `CalendarEvent/set.updated` is returned as an object map (`{ "<id>": null }`), and the shim accepts that form (and the array form) when validating updates.

## What’s still missing / next steps

1. **Search query translation beyond `in:`**
   - Current `SearchRequest` primarily resolves folder via `in:<name/path>` and ignores most other Zimbra search operators.
   - Current state: common terms now map to JMAP `Email/query.filter` (from/to/cc/bcc/subject/text, `is:read/unread/flagged`, `has:attachment`). Needs browser verification.

2. **More actions / mutations**
   - Priority, mute, and other less-common `MsgActionRequest` ops still need mapping (tagging is implemented for mail).

3. **Attachments + inline images**
   - Basic blob proxy is implemented for `GET /service/home/~/...&id=<msgId>&part=<part>` (supports attachments + inline images).
   - Still missing: multi-part downloads (ZWC “Download All”) and broader `/home/*` REST endpoints beyond blob fetch.

4. **Contacts**
   - Canonical contacts list (`/home/<user>/Contacts?fmt=cf&t=2&all=all`) and SOAP `GetContactsRequest`/`CreateContactRequest`/`ModifyContactRequest`/`ContactActionRequest` are implemented.
   - Compose autocomplete is implemented and user-verified; remaining gaps are mainly GAL behavior, contacts trash/recovery semantics, and any remaining `/home/*` Contacts REST variants.

5. **More SOAP endpoints**
   - As the UI exercises more features, implement the next missing SOAP calls as they show up in logs.

6. **Mail sending (Reply / Reply-All / New message)**
   - Next iteration: verify attachments + inline images in compose across the common flows (new/reply/reply-all, draft→send).

7. **Folder retention policy (ZWC “Retention” tab)**
   - `FolderActionRequest op="retentionpolicy"` is implemented.
   - The shim persists per-folder retention metadata and enforces `purge` rules against Stalwart by destroying messages older than the configured cutoff.
   - Enforcement is best-effort on `NoOpRequest`, plus an immediate sweep when the policy is saved.
   - Current limits:
     - `purge` is enforced; `keep` is stored but not enforced.
     - sweeps are capped by `BRIDGE_RETENTION_MAX_DESTROY_PER_SWEEP` (default `2000`) destroyed messages total per sweep across all folders. If a backlog exceeds that cap, the remainder waits for the next retention sweep window (`BRIDGE_RETENTION_SWEEP_SECS`, default `24h`).
     - folder unread/total counts refresh on the next mailbox refresh / `NoOpRequest`, not in the save response itself.
     - the retention sweep interval gate is in-memory; a bridge restart resets it and can trigger another catch-up sweep on the next active session.
   - Operator details are documented in `docs/sysadmin/SYSADMIN_RETENTION_POLICY.md`.
